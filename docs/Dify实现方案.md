## 2. 为什么使用 Dify

本项目的目标是展示 AI 产品经理能力，而不是重点展示后端工程能力。使用 Dify 可以更快完成真实 AI Agent 的搭建，并突出以下能力：

- 将用户需求拆成可执行的 AI 工作流
- 设计结构化输入字段和标准化输出结果
- 将 Prompt 从单段指令拆成多节点任务链
- 配置可运行的模型节点和节点间变量传递
- 快速验证用户输入到 AI 输出的完整闭环

## 3. App 类型选择

本项目选择 Dify 的 **Workflow App**。

原因：

- 当前产品是单轮任务型流程
- 用户输入 JD 和简历后，希望一次性获得完整诊断报告
- Workflow 更适合结构化输入、固定步骤和稳定输出
- 多节点工作流可以让每一步的 AI 判断更可控，也更方便后续做逐节点效果评测

## 4. 工作流结构

当前 Workflow 已从 MVP 三节点升级为 7 节点结构：

```text
User Input
  ↓
JD Ability Analysis
  ↓
Resume Match Diagnosis
  ↓
Weak Expression Check
  ↓
Experience Rewrite
  ↓
Risk Review & Final Report
  ↓
Output Resume Report
```

这套结构把原本一个大 Prompt 拆成 5 个 LLM 子任务，让每个节点只负责一个清晰目标。

## 5. 节点说明

### 5.1 User Input

输入字段：

| 变量名 | 类型 | 说明 |
| --- | --- | --- |
| `job_type` | Select | 岗位类型 |
| `job_description` | Paragraph | 目标岗位 JD |
| `resume_text` | Paragraph | 用户简历内容 |
| `target_experience` | Paragraph | 用户希望重点优化的经历 |

### 5.2 JD Ability Analysis

节点类型：LLM

目标：基于目标 JD 拆解岗位方向、核心能力要求、简历关键词和优化重点。

核心约束：

- 只基于 JD 内容分析
- JD 信息不足时明确说明不确定
- 输出要面向产品经理实习求职者，避免空泛总结

### 5.3 Resume Match Diagnosis

节点类型：LLM

目标：结合 JD 能力拆解和用户简历，判断简历与岗位的匹配度。

核心约束：

- 只基于简历已有内容判断
- 每个判断都要引用简历原文证据
- 本节点只做诊断，不直接改写简历

### 5.4 Weak Expression Check

节点类型：LLM

目标：定位简历中最需要优化的弱表达，例如只有执行动作、缺少目标方法结果、弱动词过多、与 JD 能力关联弱等问题。

输出包含：

- 原句
- 问题类型
- 为什么弱
- 对应 JD 能力
- 修改方向
- 优先级

### 5.5 Experience Rewrite

节点类型：LLM

目标：基于前面节点的诊断结果，生成可放入简历的项目经历表达。

核心约束：

- 不得虚构项目、职责、数据、用户数量、增长结果或业务结果
- 不得把“参与”“协助”无依据改成“主导”
- 缺少数据时只能提示用户补充真实数据
- 改写要突出产品经理能力，例如用户研究、需求分析、竞品分析、原型设计、数据分析、项目推进、AI 工具理解、Prompt 设计或效果评测

### 5.6 Risk Review & Final Report

节点类型：LLM

目标：整合前面所有节点输出，生成最终 Markdown 报告，并重点检查真实性风险。

该节点会明确提醒：

- 哪些数据需要用户确认
- 哪些职责描述不能编造
- 哪些结果表达可能过度包装
- 下一步最优先修改什么

### 5.7 Output Resume Report

节点类型：End

输出字段：

| 变量名 | 来源 |
| --- | --- |
| `resume_report` | `Risk Review & Final Report` 节点的 `text` 输出 |

## 6. 模型配置

所有 LLM 节点当前使用同一模型：

```text
gpt-4.1-mini-2025-04-14
```

Temperature：

```text
0.2
```

选择低温度的原因：

- 简历诊断需要稳定性
- 不能过度发挥
- 需要降低虚构和夸大风险
- 多节点链路中，每一步都要尽量保持可复核

## 7. 输出结构

Dify Workflow 最终输出一份 Markdown 报告，包含：

1. 岗位能力拆解
2. 简历匹配度诊断
3. 已体现能力
4. 缺失或薄弱能力
5. 简历弱表达定位
6. 项目经历改写
7. 改写理由
8. 需要用户补充的信息
9. 真实性风险提醒
10. 下一步修改建议

## 8. 关键 Prompt 设计思路

本工作流的 Prompt 不是只写一个“大而全”的指令，而是按产品流程拆成多个节点：

- JD 节点负责“理解岗位”
- 匹配节点负责“判断差距”
- 弱表达节点负责“定位问题”
- 改写节点负责“生成候选表达”
- 风险节点负责“审核和总结”

这样做的好处：

- 每个节点目标更清晰
- 输出更容易检查和评测
- 后续可以单独优化某个节点
- 面试时更容易讲清楚 AI 产品经理如何把用户任务拆成 Agent 流程

## 9. 面试讲述方式

可以这样介绍：

> 我先完成了 PM Resume Coach 的用户研究、竞品分析、PRD、Prompt 设计和 AI 效果评测。随后我用 Dify 搭建了一个多节点 Workflow App，把简历优化任务拆成 JD 能力拆解、简历匹配诊断、弱表达定位、项目经历改写、真实性风险审核和最终报告输出。这个版本可以真实输入目标 JD 和简历，并生成可解释、可复核、带风险提醒的 AI 简历优化报告。

## 10. 后续迭代方向

后续可以继续增强：

- 增加条件分支：当 JD 信息不足时，引导用户补充岗位描述
- 增加评分节点：单独输出匹配度分数、证据充分度和真实性风险分
- 增加批量运行：用多份测试简历做 AI 效果评测
- 增加知识库：沉淀产品经理岗位能力词典和优秀简历表达样例
- 接入前端原型：把 Dify API 作为真实 Agent 后端
