# Prompt 设计

## 1. 设计目标

PM Resume Coach 的 Prompt 设计目标不是让 AI “随便润色简历”，而是把产品经理实习简历优化拆成可控、可评测、可解释的 AI 任务。

本产品需要 AI 完成 5 类核心任务：

1. 拆解目标岗位 JD
2. 分析简历与 JD 的匹配度
3. 定位简历中的弱表达
4. 改写项目经历
5. 解释改写理由与真实性风险

Prompt 设计需要保证：

- 输出结构稳定
- 建议具体可执行
- 不虚构经历
- 不编造数据
- 能解释为什么这样改
- 能服务产品页面展示，而不是只生成一段聊天回复

## 2. Prompt 设计原则

### 2.1 先定义任务，再写提示词

不要一上来写“请帮我优化简历”。应先拆清楚 AI 在这个环节到底要完成什么任务。

错误示例：

```text
请帮我优化这份产品经理简历。
```

更好的方式：

```text
请基于目标岗位 JD，判断这段项目经历是否体现了用户研究、需求分析、原型设计、数据分析和项目推进能力，并指出最需要修改的 3 个问题。
```

### 2.2 输入必须结构化

AI 输出质量很大程度取决于输入质量。对于本产品，输入至少应包含：

- 岗位类型
- 目标 JD
- 简历内容
- 用户希望优化的经历
- 是否允许补充占位提示

### 2.3 输出必须结构化

产品页面需要稳定展示结果，因此 Prompt 不能只要求 AI 输出自然语言段落，而应明确输出字段。

例如：

```text
请按以下结构输出：
1. 匹配度评分
2. 已体现能力
3. 缺失能力
4. 问题段落
5. 修改建议
```

### 2.4 约束比能力更重要

简历优化场景中，AI 最大风险是过度包装或虚构经历。因此 Prompt 必须明确边界：

- 不得编造公司、项目、数据、用户数量、增长结果
- 不得把“参与”改成“主导”，除非用户原文能证明
- 缺少数据时只能提示用户补充，不能直接生成假数据
- 对可能夸大的表达必须给出风险提醒

### 2.5 Prompt 要可评测

好的 Prompt 不只是“看起来写得好”，而是可以被测试。每个 Prompt 都应有对应的评测标准，例如：

- 是否准确拆解 JD
- 是否定位到具体问题段落
- 是否给出可执行建议
- 是否避免虚构经历
- 是否输出固定字段

## 3. 总体 Prompt 架构

PM Resume Coach 可以采用“系统级规则 + 任务级 Prompt + 输出格式”的结构。

### 3.1 系统级规则

系统级规则适用于所有 AI 任务，定义产品的基本行为边界。

```text
你是 PM Resume Coach，一名面向产品经理实习求职者的 AI 简历优化助手。

你的任务是基于用户提供的目标岗位 JD 和简历内容，帮助用户理解岗位能力要求、诊断简历匹配问题，并优化项目经历表达。

你必须遵守以下规则：
1. 不得虚构用户没有提供的经历。
2. 不得编造具体数据，例如用户数、转化率、增长率、营收、DAU、GMV。
3. 如果简历中缺少数据，只能提示用户补充真实数据。
4. 不得把“参与”“协助”改写成“主导”“负责整体”，除非原文能证明用户确实主导。
5. 输出必须具体、可执行，避免泛泛建议。
6. 每次改写都必须解释改写理由。
7. 对可能过度包装或证据不足的表达，必须给出风险提醒。
8. 输出语言使用中文，风格专业、简洁、适合求职简历场景。
```

### 3.2 任务级 Prompt

任务级 Prompt 根据产品流程拆分，不建议用一个超长 Prompt 处理所有任务。

建议拆成 5 个 Prompt：

| Prompt | 使用页面 | 目标 |
| --- | --- | --- |
| JD 能力拆解 Prompt | JD 能力拆解页 | 理解目标岗位 |
| 简历匹配诊断 Prompt | 匹配度分析页 | 判断简历和 JD 是否匹配 |
| 弱表达定位 Prompt | 简历问题诊断页 | 找出具体问题段落 |
| 项目经历改写 Prompt | 项目经历改写页 | 生成优化后的简历表达 |
| 改写解释 Prompt | 改写结果页 | 解释改写逻辑与风险 |

### 3.3 输出格式

为了便于产品展示，建议每个 Prompt 都要求固定字段输出。真实开发中可进一步使用 JSON 结构；在作品集阶段，可以先用 Markdown 表格或字段列表展示。

## 4. Prompt 一：JD 能力拆解

### 使用场景

用户粘贴目标岗位 JD 后，系统需要拆解岗位方向、核心能力、关键词和简历表达重点。

### 输入变量

```text
岗位类型：{{job_type}}
目标岗位 JD：{{job_description}}
```

### Prompt 模板

```text
请基于以下目标岗位 JD，拆解该产品经理实习岗位的核心能力要求。

岗位类型：
{{job_type}}

目标岗位 JD：
{{job_description}}

请遵守以下规则：
1. 只基于 JD 内容分析，不要引入无关岗位要求。
2. 如果 JD 信息不足，请说明“不确定”的部分。
3. 输出要面向产品经理实习求职者，语言清晰易懂。
4. 不要生成简历改写内容，本步骤只做岗位能力拆解。

请按以下结构输出：

## 岗位方向判断
- 判断该岗位更偏向：AI 产品 / 用户产品 / 增长产品 / 内容产品 / 平台产品 / 其他
- 判断理由：

## 核心能力要求
请输出 3-6 项能力，每项包括：
- 能力名称：
- JD 依据：
- 能力解释：
- 简历中应体现的证据：
- 优先级：高 / 中 / 低

## 简历关键词
请输出适合在简历中体现的 8-12 个关键词。

## 简历优化提醒
请告诉用户后续优化简历时最应该突出什么。
```

### 输出示例

```text
## 岗位方向判断
- 方向：AI 产品
- 判断理由：JD 中多次提到 AI 工具、Prompt、效果评测和用户场景，因此岗位更关注 AI 产品能力。

## 核心能力要求
1. AI 场景理解
- JD 依据：负责分析用户在 AI 工具中的使用需求
- 能力解释：能判断 AI 能解决什么用户问题，以及适合放在哪个使用流程中
- 简历中应体现的证据：AI 工具体验、用户调研、场景分析、功能设计
- 优先级：高
```

### 评测标准

- 是否准确判断岗位方向
- 是否引用 JD 中的真实依据
- 是否拆出清晰能力，而不是复述 JD
- 是否给出简历中应体现的证据

## 5. Prompt 二：简历匹配诊断

### 使用场景

用户输入简历后，系统基于前一步的 JD 能力拆解，判断简历和岗位的匹配程度。

### 输入变量

```text
岗位类型：{{job_type}}
JD 能力拆解结果：{{jd_analysis}}
简历内容：{{resume_text}}
```

### Prompt 模板

```text
请基于目标岗位能力要求，诊断用户简历与岗位 JD 的匹配程度。

岗位类型：
{{job_type}}

JD 能力拆解结果：
{{jd_analysis}}

用户简历：
{{resume_text}}

请遵守以下规则：
1. 只基于用户简历中已有内容判断，不要猜测用户做过什么。
2. 不要改写简历，本步骤只做匹配诊断。
3. 如果某项能力没有证据，请明确说明“简历中暂未体现”。
4. 评分必须给出理由，不能只给分数。

请按以下结构输出：

## 综合匹配度
- 分数：0-100
- 结论：高匹配 / 中匹配 / 低匹配
- 核心理由：

## 已体现能力
请列出 2-5 项，每项包括：
- 能力：
- 简历证据：
- 与 JD 的匹配说明：

## 缺失或薄弱能力
请列出 2-5 项，每项包括：
- 能力：
- 问题说明：
- 建议补充的真实证据：

## 修改优先级
请按优先级列出最应该修改的 3 个问题。
```

### 评测标准

- 是否根据 JD 能力拆解进行诊断
- 是否引用简历原文作为证据
- 是否明确区分已体现能力和缺失能力
- 是否避免无依据夸奖用户

## 6. Prompt 三：弱表达定位

### 使用场景

系统在简历诊断页标注具体问题段落，帮助用户知道哪里需要改。

### 输入变量

```text
JD 能力拆解结果：{{jd_analysis}}
简历内容：{{resume_text}}
```

### Prompt 模板

```text
请找出用户简历中最需要优化的弱表达，并说明问题原因。

JD 能力拆解结果：
{{jd_analysis}}

用户简历：
{{resume_text}}

请重点识别以下问题：
1. 只有执行动作，没有目标、方法或结果。
2. 使用“参与”“协助”“整理”等弱动词，但没有说明个人贡献。
3. 缺少用户视角、需求分析、产品方案或数据结果。
4. 与目标 JD 的关键能力关联较弱。
5. 表达过泛，无法体现产品经理能力。

请按以下结构输出 3-6 条：

## 弱表达列表

### 问题 {{序号}}
- 原句：
- 问题类型：弱动词 / 缺少目标 / 缺少结果 / 缺少用户视角 / 缺少数据 / 岗位相关性弱
- 问题说明：
- 对应 JD 能力：
- 修改方向：
- 优先级：高 / 中 / 低
```

### 评测标准

- 是否定位到具体原句
- 是否说明为什么弱
- 是否能对应到 JD 能力
- 修改方向是否具体

## 7. Prompt 四：项目经历改写

### 使用场景

用户选择某段项目经历后，系统生成可放入简历的优化版本。

### 输入变量

```text
岗位类型：{{job_type}}
JD 能力拆解结果：{{jd_analysis}}
原始项目经历：{{project_experience}}
弱表达诊断：{{weak_expression_analysis}}
用户补充信息：{{user_additional_info}}
```

### Prompt 模板

```text
请基于目标岗位 JD 和用户原始经历，改写这段产品经理实习简历项目经历。

岗位类型：
{{job_type}}

JD 能力拆解结果：
{{jd_analysis}}

原始项目经历：
{{project_experience}}

弱表达诊断：
{{weak_expression_analysis}}

用户补充信息：
{{user_additional_info}}

请严格遵守以下规则：
1. 不得虚构项目、职责、数据、用户数量、增长结果或业务结果。
2. 不得把“参与”“协助”直接改成“主导”，除非原文或补充信息能证明。
3. 如果缺少数据，请使用“建议补充真实数据：……”提示，不要编造。
4. 改写应突出产品经理能力，例如用户研究、需求分析、竞品分析、原型设计、数据分析、项目推进。
5. 改写结果要适合放入简历，语言简洁，建议 1-3 条 bullet。
6. 改写后必须解释每条改动的理由。

请按以下结构输出：

## 优化版本
- bullet 1
- bullet 2
- bullet 3

## 改写理由
请逐条说明为什么这样改。

## 需要用户补充的信息
列出缺少但会显著提升简历质量的信息。

## 风险提醒
指出是否存在可能过度包装、证据不足或需要用户确认真实性的地方。
```

### 输出示例

```text
## 优化版本
- 围绕校园二手交易场景，访谈 8 名学生用户并归纳交易信任、商品筛选、沟通效率 3 类核心需求，输出需求优先级和核心交易流程。
- 协助设计发布商品、搜索筛选、站内沟通等关键页面原型，并根据用户反馈调整信息展示和操作路径。

## 改写理由
- 第一条补充了目标场景、调研对象、需求归纳和流程设计，更能体现用户研究与需求分析能力。
- 第二条把“协助完成原型设计”具体化为页面和优化方向，更接近产品经理简历表达。

## 需要用户补充的信息
- 如果有问卷数量、原型测试人数、功能点击率或用户满意度，可补充真实数据。

## 风险提醒
- 请确认“8 名学生用户”和“3 类核心需求”是否真实。如果没有准确数据，应替换为真实数量或更保守表达。
```

### 评测标准

- 是否保持经历真实
- 是否比原文更能体现产品能力
- 是否适合直接放入简历
- 是否有改写理由
- 是否提醒数据真实性风险

## 8. Prompt 五：改写解释与学习反馈

### 使用场景

用户看到改写结果后，系统解释这次改写具体提升了什么，帮助用户学习产品经理简历表达方法。

### 输入变量

```text
原始表达：{{original_text}}
优化版本：{{rewritten_text}}
JD 能力拆解结果：{{jd_analysis}}
```

### Prompt 模板

```text
请对比原始表达和优化版本，解释本次改写为什么更适合产品经理实习岗位。

原始表达：
{{original_text}}

优化版本：
{{rewritten_text}}

JD 能力拆解结果：
{{jd_analysis}}

请按以下结构输出：

## 改写前的问题
列出 2-4 个具体问题。

## 改写后的提升
说明优化版本在哪些方面更符合产品经理简历表达。

## 对应的产品能力
列出优化版本体现了哪些产品经理能力。

## 用户可迁移的方法
用一句话告诉用户以后写类似经历时应遵循什么方法。
```

### 评测标准

- 是否能解释清楚改写逻辑
- 是否帮助用户理解产品表达方法
- 是否避免空泛夸奖
- 是否能对应 JD 能力

## 11. Prompt 评测标准

### 11.1 准确性

AI 是否正确理解 JD 和简历内容。

评分标准：

- 5 分：准确理解，能引用原文依据
- 3 分：基本理解，但有遗漏
- 1 分：误解关键信息

### 11.2 具体性

AI 是否给出具体问题和修改建议。

评分标准：

- 5 分：定位到具体段落，并给出可执行建议
- 3 分：有建议，但不够具体
- 1 分：只给泛泛建议

### 11.3 可用性

改写结果是否能被用户直接或稍作修改后放入简历。

评分标准：

- 5 分：基本可直接使用
- 3 分：需要明显修改
- 1 分：不可使用

### 11.4 真实性

AI 是否避免虚构经历或编造数据。

评分标准：

- 5 分：没有虚构，并主动提醒补充真实信息
- 3 分：有轻微过度包装风险
- 1 分：出现明显虚构

### 11.5 结构稳定性

AI 是否按要求输出固定字段。

评分标准：

- 5 分：完全符合结构
- 3 分：结构基本符合，但有缺失
- 1 分：输出混乱

## 12. Prompt 迭代方法

Prompt 设计不是一次写完，而是通过测试不断迭代。

### 12.1 发现问题

常见问题包括：

- 输出太泛
- 输出太长
- 没有引用 JD 或简历依据
- 改写出现虚构数据
- 结构不稳定
- 评分缺少理由

### 12.2 修改 Prompt

根据问题调整 Prompt：

| 问题 | 修改方式 |
| --- | --- |
| 输出太泛 | 要求引用原文，并输出具体段落 |
| 输出太长 | 限制每部分字数或 bullet 数量 |
| 虚构数据 | 增加强约束：缺少数据只能提示补充 |
| 结构不稳定 | 明确字段格式，必要时使用 JSON |
| 诊断不准 | 增加 JD 能力拆解作为上下文 |
| 改写太夸张 | 要求保留原始职责边界 |

### 12.3 回归测试

每次修改 Prompt 后，用同一批测试样例重新测试，观察是否解决旧问题，同时是否引入新问题。

## 13. 给产品经理的 Prompt 设计方法

产品经理做 Prompt 设计时，可以按 6 步走：

### 第一步：定义用户任务

先写清用户到底要完成什么任务。

例如：

```text
用户想知道自己的简历是否匹配某个 AI 产品实习岗位。
```

### 第二步：拆分 AI 子任务

不要让一个 Prompt 做所有事，而是拆成多个子任务。

例如：

```text
拆 JD → 诊断简历 → 定位问题 → 改写经历 → 解释理由
```

### 第三步：明确输入

列出 AI 需要哪些信息。

例如：

```text
岗位类型、JD、简历、项目经历、用户补充数据。
```

### 第四步：明确输出

定义输出字段，而不是只写“请分析一下”。

例如：

```text
匹配度评分、已体现能力、缺失能力、问题段落、修改建议。
```

### 第五步：写清边界

告诉 AI 不能做什么。

例如：

```text
不能编造数据，不能虚构经历，不能夸大用户职责。
```

### 第六步：设计评测

用测试样例判断 Prompt 好不好。

例如：

```text
准确性、具体性、可用性、真实性、结构稳定性。
```
