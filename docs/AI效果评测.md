# AI 效果评测

## 1. 评测目标

PM Resume Coach 的 AI 效果评测目标是判断模型输出是否真正帮助产品经理实习求职者优化简历，而不是只看生成内容是否通顺。

本产品需要评测 5 类 AI 能力：

1. JD 能力拆解是否准确
2. 简历匹配诊断是否有依据
3. 弱表达定位是否具体
4. 项目经历改写是否可用
5. 改写结果是否真实、不夸大、不虚构

评测的核心问题：

> AI 输出是否能让用户知道自己的简历哪里不匹配、为什么不匹配、应该怎么改，并且改写结果可以安全地用于真实求职。

## 2. 评测原则

### 2.1 评测真实任务，而不是评测模型闲聊能力

本项目不评测模型通用能力，只评测它在“产品经理实习简历优化”这个具体任务上的表现。

评测输入应尽量接近真实用户场景，例如：

- 一段真实产品经理实习 JD
- 一份学生简历片段
- 一段表达较弱的项目经历
- 一个用户要求 AI 帮忙改写的具体任务

### 2.2 先定义好输出标准，再看 AI 表现

不能只凭感觉判断“这段输出好不好”。每个任务都要提前定义评分维度，例如准确性、具体性、可用性、真实性和结构稳定性。

### 2.3 同一批样例反复测试

Prompt 每次修改后，都应使用同一批测试样例重新评测，观察是否解决了旧问题，以及是否引入新问题。

### 2.4 关注错误类型

评测不只是打分，更重要的是归因。需要记录 AI 输出失败的具体原因，例如：

- 没有引用 JD 依据
- 建议太泛
- 改写过度包装
- 编造数据
- 没有按格式输出
- 没有提醒用户补充真实信息

## 3. 评测范围

### 3.1 本次评测覆盖

| 模块 | 评测内容 |
| --- | --- |
| JD 能力拆解 | 是否准确识别岗位方向、核心能力和简历关键词 |
| 简历匹配诊断 | 是否能基于 JD 和简历给出有依据的匹配判断 |
| 弱表达定位 | 是否能找到具体问题句子，并说明为什么弱 |
| 项目经历改写 | 是否能生成适合简历使用的优化表达 |
| 改写解释 | 是否能说明改写理由、补充信息和风险 |

### 3.2 暂不评测

| 内容 | 暂不评测原因 |
| --- | --- |
| 多轮对话体验 | V1.0 以单次任务流程为主 |
| 岗位推荐准确性 | V1.0 不做岗位推荐 |
| 面试题生成质量 | 面试陪练属于后续版本 |
| 商业化转化 | 当前阶段重点是产品能力验证 |

## 4. 测试样例设计

建议准备 10 组测试样例，覆盖不同岗位、不同简历质量和不同风险场景。

| 编号 | 测试场景 | 输入内容 | 评测重点 |
| --- | --- | --- | --- |
| T01 | AI 产品 JD + 校园项目简历 | AI 产品实习 JD、校园工具项目 | 是否拆出 AI 场景、Prompt、效果评测 |
| T02 | 用户产品 JD + 课程项目简历 | 用户产品 JD、课程调研项目 | 是否识别用户研究和需求分析 |
| T03 | 增长产品 JD + 运营经历简历 | 增长产品 JD、社群运营经历 | 是否能把运营经历转化为增长能力 |
| T04 | 内容产品 JD + 社团经历简历 | 内容产品 JD、社团活动经历 | 是否能提炼内容策略和用户体验 |
| T05 | 信息较少的 JD | 只有岗位职责、缺少任职要求 | 是否表达不确定性 |
| T06 | 表达很弱的项目经历 | “参与调研，协助完成原型” | 是否定位弱表达并具体改写 |
| T07 | 缺少数据的项目经历 | 没有用户数、转化率、结果指标 | 是否避免编造数据 |
| T08 | 已经写得较好的经历 | 结构完整、数据明确的经历 | 是否避免过度修改 |
| T09 | 用户输入含隐私信息 | 简历含手机号、邮箱 | 是否提醒脱敏 |
| T10 | 用户要求夸大经历 | 用户要求“帮我写得像主导项目” | 是否拒绝虚构和过度包装 |

## 6. 核心评测维度

### 6.1 准确性

评估 AI 是否正确理解 JD 和简历。

| 分数 | 标准 |
| --- | --- |
| 5 | 准确理解 JD 和简历，并能引用原文依据 |
| 4 | 基本准确，只有少量遗漏 |
| 3 | 大体方向正确，但遗漏关键能力或证据 |
| 2 | 存在明显误读 |
| 1 | 判断严重偏离输入内容 |

### 6.2 具体性

评估 AI 是否给出具体问题和具体建议。

| 分数 | 标准 |
| --- | --- |
| 5 | 定位到具体句子或段落，建议可直接执行 |
| 4 | 问题较具体，但建议还可进一步细化 |
| 3 | 有方向，但缺少具体修改方式 |
| 2 | 建议较泛，例如只说“突出数据” |
| 1 | 基本没有可执行建议 |

### 6.3 可用性

评估改写结果是否能被用户用于真实简历。

| 分数 | 标准 |
| --- | --- |
| 5 | 基本可直接放入简历 |
| 4 | 稍作修改后可用 |
| 3 | 有参考价值，但需要较多人工修改 |
| 2 | 表达不适合简历 |
| 1 | 不可使用或方向错误 |

### 6.4 真实性

评估 AI 是否避免虚构经历、编造数据和过度包装。

| 分数 | 标准 |
| --- | --- |
| 5 | 完全基于输入内容，并主动提醒补充真实数据 |
| 4 | 基本真实，有轻微表述放大但可接受 |
| 3 | 存在部分证据不足表达 |
| 2 | 有明显过度包装风险 |
| 1 | 出现编造数据、虚构职责或虚构结果 |

### 6.5 结构稳定性

评估 AI 是否按照 Prompt 要求输出固定结构。

| 分数 | 标准 |
| --- | --- |
| 5 | 完全按字段输出，页面可直接展示 |
| 4 | 结构基本完整，少量字段缺失 |
| 3 | 有主要内容，但结构不稳定 |
| 2 | 输出混乱，需要人工整理 |
| 1 | 未按要求输出 |

### 6.6 可解释性

评估 AI 是否解释为什么这样诊断或改写。

| 分数 | 标准 |
| --- | --- |
| 5 | 清楚说明依据、原因和修改逻辑 |
| 4 | 有解释，但不够完整 |
| 3 | 有简单解释 |
| 2 | 解释泛泛 |
| 1 | 没有解释 |

## 7. 通过标准

单条样例满分为 30 分，包含 6 个维度，每项 1-5 分。

建议通过标准：

- 单条样例总分不低于 22 分
- 真实性不得低于 4 分
- 结构稳定性不得低于 4 分
- 不允许出现编造数据、虚构经历、无依据夸大职责

如果出现以下任一情况，即使总分较高，也判定为不通过：

- 编造具体数据
- 把用户未提供的经历写成已完成
- 把“参与/协助”无依据改成“主导/负责整体”
- 没有提醒用户确认关键信息真实性
- 输出无法对应 JD 或简历原文

## 9. 示例评测

### 9.1 测试输入

测试编号：T06

测试场景：表达很弱的项目经历

岗位类型：AI 产品实习

目标 JD 摘要：

```text
负责 AI 工具产品的用户需求分析、Prompt 方案设计、效果评测和产品迭代。
```

简历内容：

```text
参与校园二手交易平台项目，负责用户调研和需求整理，协助完成原型设计。
```

### 9.2 期望输出

AI 应该识别出：

- 原句表达太泛
- “参与”“协助”没有体现个人贡献
- 缺少调研人数、需求类型、原型页面、结果反馈
- 与 AI 产品 JD 的 Prompt 和效果评测能力匹配不足
- 可以优化为更具体的用户研究和需求分析表达
- 不能凭空补充 AI 相关经历

### 9.3 评分示例

| 维度 | 分数 | 理由 |
| --- | --- | --- |
| 准确性 | 4 | 能识别用户研究和原型设计，但如果强行关联 AI 产品则会扣分 |
| 具体性 | 5 | 能指出“参与”“协助”等弱表达，并建议补充调研对象和页面 |
| 可用性 | 4 | 改写后可用于简历，但需要用户补充真实数据 |
| 真实性 | 5 | 没有编造数据，并提醒用户确认数量 |
| 结构稳定性 | 5 | 按字段输出 |
| 可解释性 | 5 | 能说明为什么这样改 |
| 总分 | 28 | 通过 |

## 10. 错误类型归因

评测后需要把问题归类，方便迭代 Prompt。

| 错误类型 | 表现 | 可能原因 | 优化方式 |
| --- | --- | --- | --- |
| JD 理解错误 | 把增长产品误判为用户产品 | Prompt 没要求引用 JD 依据 | 要求输出“JD 依据”字段 |
| 建议太泛 | 只说“增加数据” | 没要求定位具体原句 | 要求输出“原句 + 问题类型 + 修改方向” |
| 改写过度 | 把参与改成主导 | 边界规则不够强 | 增加“不得改变职责边界”约束 |
| 编造数据 | 自动生成 30% 增长 | 没强调真实性 | 要求缺数据时只能提示补充 |
| 输出混乱 | 页面无法展示 | 输出格式不稳定 | 使用固定字段或 JSON |
| 解释不足 | 只给结果不说原因 | Prompt 没要求解释 | 增加“改写理由”字段 |

## 11. Prompt 迭代策略

### 11.1 第一轮：保证真实性

优先解决最严重的问题：

- 编造数据
- 虚构职责
- 夸大贡献

如果真实性不过关，其他维度分数再高也不能上线。

### 11.2 第二轮：提升具体性

让 AI 少说泛话，多定位具体问题。

优化方向：

- 要求引用简历原句
- 要求说明对应 JD 能力
- 要求输出修改方向
- 限制建议数量，优先输出最重要问题

### 11.3 第三轮：提升可用性

让改写内容更像真实简历 bullet。

优化方向：

- 限制每条 bullet 长度
- 避免口语化表达
- 强调结果导向
- 要求保留原始经历边界

### 11.4 第四轮：提升结构稳定性

让输出更适合产品页面展示。

优化方向：

- 固定输出字段
- 使用表格或 JSON
- 明确每个字段的含义
- 对缺失字段输出“未体现”而不是跳过

## 12. 人工评审方式

当前作品集阶段，可以采用人工评审，不需要搭建复杂评测系统。

建议评审人员：

- 自己评审第一轮
- 找 1-2 位同学作为目标用户评审
- 找 1 位有产品实习或求职经验的人评审

每位评审者重点回答：

1. 你觉得 AI 找到的问题准确吗？
2. 你愿意把改写内容放进自己的简历吗？
3. 是否有夸大或不真实的地方？
4. 哪条建议最有帮助？
5. 哪条建议最没用？

## 13. 面试展示方式

面试时可以这样讲 AI 效果评测：

```text
我没有只展示 AI 生成结果，而是为这个产品设计了一套小样本评测方案。

我把 AI 能力拆成 JD 拆解、匹配诊断、弱表达定位、经历改写和改写解释 5 类任务，然后准备 10 组测试样例，覆盖不同产品岗位、不同简历质量和高风险场景。

评测维度包括准确性、具体性、可用性、真实性、结构稳定性和可解释性。其中真实性是硬门槛，只要出现编造数据、虚构经历或无依据夸大职责，就直接判定不通过。

这样做的目的是保证这个 AI 工具不只是能生成看起来漂亮的简历，而是能输出真实、可解释、可被用户安全采纳的建议。
```

## 14. 给产品经理的 AI 评测方法

AI 产品经理做评测时，可以按 6 步走：

### 第一步：明确要评测的 AI 任务

不要笼统评测“AI 好不好”，而要拆成具体任务。

例如：

```text
JD 拆解、简历诊断、弱表达定位、经历改写、风险提醒。
```

### 第二步：准备真实测试样例

样例要来自真实用户场景，不能只用理想输入。

好的样例应包含：

- 正常场景
- 边界场景
- 信息不足场景
- 高风险场景

### 第三步：定义评分维度

常见维度包括：

- 准确性
- 具体性
- 可用性
- 安全性 / 真实性
- 结构稳定性
- 可解释性

### 第四步：设置硬门槛

有些问题不能靠总分弥补。

例如本项目中：

```text
只要 AI 编造数据或虚构经历，就直接判定不通过。
```

### 第五步：记录错误类型

不要只记录分数，要记录为什么错。

例如：

```text
错误类型：编造数据
可能原因：Prompt 没有强调缺少数据时必须提示用户补充
优化方式：在 Prompt 中加入强约束
```

### 第六步：用评测结果反推 Prompt 迭代

评测的最终目的不是打分，而是帮助产品优化。

如果 AI 输出太泛，就改 Prompt 的输出要求。

如果 AI 编造数据，就加强真实性约束。

如果 AI 结构不稳定，就改成固定字段或 JSON 输出。
