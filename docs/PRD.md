## 2. 项目背景

产品经理实习求职者在准备简历时，常见问题不是完全没有经历，而是不知道如何把已有经历表达成岗位需要的产品能力。很多用户会把课程项目、校园活动、比赛经历或短期实习写成执行记录，例如“参与调研”“整理资料”“协助完成项目”，但这些表述很难体现用户研究、需求分析、方案设计、数据分析、项目推进等产品经理核心能力。

同时，不同产品经理实习岗位的 JD 侧重点不同。AI 产品更关注 AI 工具理解、Prompt 设计和效果评测；增长产品更关注数据分析、转化路径和实验意识；用户产品更关注用户洞察、体验设计和需求判断。用户如果用同一份简历投递多个岗位，容易出现“经历不错，但岗位匹配度不明显”的问题。

现有通用 AI 工具可以帮助用户润色简历，但用户需要自己设计 Prompt，也很难判断输出是否真的符合目标岗位。传统简历工具更擅长模板、排版和基础诊断，但对产品经理实习场景中的 JD 拆解、岗位匹配和项目经历表达支持有限。

因此，PM Resume Coach 希望聚焦“产品经理实习求职者”这一细分人群，提供基于目标 JD 的简历匹配诊断和项目经历改写能力。

## 3. 产品目标

### 3.1 用户目标

- 帮助用户理解目标岗位 JD 背后的能力要求
- 帮助用户判断当前简历与目标岗位的匹配程度
- 帮助用户发现简历中表达较弱、缺少产品能力证明的内容
- 帮助用户把普通经历改写成更符合产品经理岗位要求的简历表达
- 帮助用户避免 AI 过度包装、虚构经历或编造数据

### 3.2 产品目标

- 建立一条从 JD 输入到简历改写的完整闭环
- 将复杂 Prompt 封装为简单、可操作的产品流程
- 输出结构化、可解释、可复制的简历优化建议
- 为后续面试陪练、岗位版本管理、作品集优化等功能留出扩展空间

### 3.3 暂不追求的目标

- 不做通用简历模板平台
- 不做完整招聘平台
- 不做岗位推荐系统
- 不做自动投递工具
- 不做完全替代用户判断的简历生成器

## 4. 目标用户

### 4.1 核心用户

正在寻找产品经理实习机会的大学生、研究生或应届生。

### 4.2 用户特征

- 有课程项目、校园项目、社团活动、比赛经历、个人项目或短期实习经历
- 对产品经理岗位能力要求理解不深
- 不清楚自己的简历是否匹配目标岗位
- 希望获得具体、可执行的简历修改建议
- 可能尝试过用 ChatGPT、Kimi、豆包等 AI 工具改简历，但觉得建议偏泛

### 4.3 典型使用场景

用户看到一个产品经理实习岗位，准备投递前，希望快速判断自己的简历是否匹配该岗位，并优化其中 1-2 段最关键的项目经历。

## 5. 用户痛点

### 痛点一：看不懂 JD 背后的产品能力要求

用户能读懂 JD 字面信息，但无法拆解背后的能力模型。例如“负责需求分析”对应用户调研、场景拆解、需求优先级判断和方案设计，而不仅仅是“写需求”。

### 痛点二：经历存在，但表达不像产品经理

用户简历中常出现“参与”“协助”“负责整理”等弱表达，缺少目标用户、问题、方案、结果和个人贡献，导致经历看起来像执行记录，而不是产品能力证明。

### 痛点三：不会根据不同岗位调整简历

用户常用同一份简历投递 AI 产品、增长产品、用户产品等不同岗位，但这些岗位的能力侧重点不同。如果简历不调整，容易降低匹配度。

### 痛点四：通用 AI 建议太泛，难以直接使用

通用 AI 工具可以润色文字，但经常给出“突出成果”“增加数据”等泛化建议，缺少逐段诊断、改写理由和真实性风险提醒。

## 6. 产品定位

PM Resume Coach 是一款面向产品经理实习求职者的 AI 简历优化助手。

一句话定位：

> 基于目标岗位 JD，帮助产品经理实习求职者诊断简历匹配度，并将已有经历改写成更符合岗位要求的产品表达。

核心差异点：

- 不是泛简历润色，而是聚焦产品经理实习场景
- 不是只改文字，而是先拆解 JD、再诊断匹配、最后改写经历
- 不是黑盒生成，而是解释为什么这样改，并提醒用户补充真实信息

### 7.2 V1.0 暂不做功能

| 功能 | 暂不做原因 |
| --- | --- |

| 社区分享 | 初期用户规模不足，且会增加内容审核成本 |

## 8. 核心用户流程

### 8.1 主流程

1. 用户进入产品首页
2. 用户选择目标岗位类型，例如 AI 产品、增长产品、用户产品、内容产品
3. 用户粘贴目标岗位 JD
4. 系统展示 JD 能力拆解结果
5. 用户粘贴简历内容或核心项目经历
6. 系统输出简历与 JD 的匹配度诊断
7. 系统标注简历中的弱表达和缺失能力证据
8. 用户选择需要优化的项目经历
9. 系统生成改写版本
10. 系统解释改写理由和真实性风险
11. 用户复制改写内容，并回到自己的简历中修改

### 8.2 流程图

```text
进入首页
  ↓
选择岗位类型
  ↓
粘贴目标 JD
  ↓
AI 拆解岗位能力
  ↓
粘贴简历内容
  ↓
AI 输出匹配诊断
  ↓
选择要优化的经历
  ↓
AI 生成改写建议
  ↓
查看改写理由
  ↓
复制并修改简历
```

## 9. 功能需求

### 9.1 岗位类型选择

#### 功能说明

用户在开始分析前选择目标岗位类型，帮助 AI 使用更贴近场景的能力模型。

#### 可选类型

- AI 产品
- 用户产品
- 增长产品
- 内容产品
- 平台产品
- 暂不确定

#### 输入

用户点击选择一个岗位类型。

#### 输出

系统记录岗位类型，并在后续 JD 拆解和简历诊断中作为上下文。

#### 异常状态

- 用户未选择岗位类型：允许继续，默认按“通用产品经理实习”处理
- 用户选择“暂不确定”：系统提示后续将根据 JD 自动判断侧重点

### 9.2 JD 输入

#### 功能说明

用户粘贴目标岗位 JD，作为后续能力拆解和简历匹配的依据。

#### 输入要求

- 支持纯文本粘贴
- 建议包含岗位职责和任职要求
- 最少 50 字
- 最多 5000 字

#### 输出

进入 JD 拆解模块。

#### 异常状态

| 异常 | 系统处理 |
| --- | --- |
| 内容为空 | 提示用户粘贴岗位 JD |
| 字数过少 | 提示“当前 JD 信息较少，分析可能不完整” |
| 内容明显不是 JD | 提示用户检查输入内容 |
| 内容过长 | 提示用户删除无关公司介绍或福利信息 |

### 9.3 JD 能力拆解

#### 功能说明

AI 根据用户输入的 JD，拆解出该岗位重点关注的产品能力。

#### 输出结构

| 字段 | 说明 |
| --- | --- |
| 岗位方向判断 | 判断该 JD 更偏 AI 产品、增长产品、用户产品等 |
| 核心能力要求 | 提炼 3-6 个关键能力 |
| 能力解释 | 用求职者能理解的语言解释每个能力 |
| 简历关键词 | 提供适合在简历中体现的关键词 |
| 优先级 | 标注哪些能力最重要 |

#### 示例输出

```text
岗位方向：AI 产品实习

核心能力：
1. 用户场景理解：能发现 AI 工具在具体用户任务中的使用价值
2. Prompt 与效果评测：能设计提示词，并判断 AI 输出是否可用
3. 需求分析：能把用户问题转化为产品功能
4. 协作推进：能与设计、研发或运营协作推动方案落地
```

### 9.4 简历输入

#### 功能说明

用户粘贴简历全文，或只粘贴希望优化的项目经历。

#### 输入要求

- 支持纯文本
- 建议用户先删除手机号、邮箱、身份证号等隐私信息
- 最少 100 字
- 最多 8000 字

#### 隐私提示

输入框上方展示提示：

```text
建议先删除姓名、手机号、邮箱、学校证件号等敏感信息，再进行分析。
```

#### 异常状态

| 异常 | 系统处理 |
| --- | --- |
| 内容为空 | 提示用户粘贴简历内容 |
| 内容过短 | 提示分析可能不完整 |
| 检测到手机号或邮箱 | 提醒用户可先脱敏再提交 |

### 9.5 简历匹配诊断

#### 功能说明

AI 将简历内容与 JD 能力要求进行对比，输出匹配度诊断。

#### 输出结构

| 字段 | 说明 |
| --- | --- |
| 综合匹配度 | 0-100 分 |
| 匹配结论 | 高匹配 / 中匹配 / 低匹配 |
| 已体现能力 | 简历中已体现的岗位能力 |
| 缺失能力 | JD 要求但简历中体现不足的能力 |
| 证据段落 | 简历中对应能力的原文 |
| 修改优先级 | 哪些问题应优先修改 |

#### 评分规则

| 分数区间 | 说明 |
| --- | --- |
| 80-100 | 简历与岗位高度匹配，只需优化表达 |
| 60-79 | 有一定匹配基础，但需要强化关键能力 |
| 40-59 | 部分经历相关，但岗位能力体现不足 |
| 0-39 | 当前简历与 JD 匹配度较低，需要补充经历或调整投递方向 |

### 9.6 弱表达定位

#### 功能说明

AI 标出简历中表达较弱、信息不足或不符合 PM 岗位语言的句子。

#### 识别规则

重点识别以下问题：

- 只写执行动作，没有目标和结果
- 使用“参与”“协助”等弱动词，但没有说明个人贡献
- 缺少用户、场景、问题、方案、数据、结果
- 与目标 JD 关键词无关
- 表达过泛，无法体现产品能力

#### 输出结构

| 字段 | 说明 |
| --- | --- |
| 原句 | 简历中的原始表达 |
| 问题类型 | 弱动词 / 缺少结果 / 缺少用户视角 / 缺少数据等 |
| 问题说明 | 为什么这句话不够有竞争力 |
| 修改方向 | 应该补充或突出什么信息 |

### 9.7 项目经历改写

#### 功能说明

用户选择一段项目经历后，AI 根据 JD 要求和诊断结果生成改写版本。

#### 改写原则

- 基于用户已有信息改写
- 不编造用户没有提供的经历
- 不编造具体数据
- 缺少数据时使用占位提示，例如“可补充转化率/用户数/效率提升数据”
- 优先体现产品经理能力
- 保持简历语言简洁、结果导向

#### 输出结构

| 字段 | 说明 |
| --- | --- |
| 原始表达 | 用户输入的原文 |
| 优化版本 | AI 改写后的简历表达 |
| 改写理由 | 为什么这样改 |
| 需用户补充 | 哪些信息需要用户补充真实数据 |
| 风险提醒 | 是否存在表述过度、证据不足等问题 |

#### 示例

原始表达：

```text
参与校园二手交易平台项目，负责用户调研和需求整理，协助完成原型设计。
```

优化版本：

```text
围绕校园二手交易场景，访谈 8 名学生用户并整理 3 类核心需求，输出需求优先级与核心交易流程，协助完成发布商品、搜索筛选、站内沟通等关键页面原型设计。
```

改写理由：

```text
相比原句，新版本补充了目标场景、调研对象、需求归纳、流程设计和具体页面，更能体现用户研究、需求分析和原型设计能力。
```

风险提醒：

```text
请确认“8 名学生用户”和“3 类核心需求”是否真实。如果没有准确数据，应替换为真实数量或改为“多名用户”“若干类需求”。
```

### 9.8 改写结果复制

#### 功能说明

用户可以一键复制优化后的项目经历表达。

#### 交互要求

- 复制成功后显示“已复制”
- 保留改写理由，方便用户理解和二次修改
- 不自动覆盖用户原简历

## 11. AI 输出规则

### 11.1 基本原则

AI 输出必须遵守以下规则：

1. 不虚构用户经历
2. 不编造具体数据
3. 不把用户未做过的事情写成已完成
4. 缺少信息时提示用户补充
5. 所有改写都要解释理由
6. 输出要结构化，避免大段泛泛建议
7. 对可能过度包装的内容进行风险提醒

### 11.2 禁止输出

以下内容不允许直接生成：

- 编造的公司名、岗位名、项目名
- 编造的用户数、转化率、GMV、DAU 等数据
- “主导完整产品从 0 到 1 落地”等与用户输入不符的夸大表达
- 无证据的能力判断，例如“具备优秀商业洞察力”

### 11.3 推荐输出方式

当用户缺少数据时，AI 应输出：

```text
建议补充真实数据：例如调研人数、问卷回收量、功能点击率、用户反馈数量、效率提升比例等。
```

而不是直接编造：

```text
提升用户转化率 30%。
```

### 11.4 输出风格

- 简洁
- 具体
- 结果导向
- 符合产品经理简历表达
- 保留学生求职者的真实经历边界

| 虚构风险率 | 输出中出现编造或过度包装内容的比例 |
| 诊断准确率 | 用户认为 AI 找到的问题是否准确 |
| 改写解释有效性 | 用户是否理解为什么这样改 |

### 12.3 MVP 验证目标

第一版可以用小样本验证：

- 5-8 位目标用户试用
- 80% 用户能完成从 JD 输入到改写复制的主流程
- 60% 以上用户认为改写建议比通用 AI 更具体
- 0 个明显编造经历或数据的高风险输出

## 13. 异常状态与边界

### 13.1 输入异常

| 场景 | 处理方式 |
| --- | --- |
| JD 为空 | 提示用户粘贴岗位 JD |
| JD 信息过少 | 提示分析可能不完整 |
| 简历为空 | 提示用户粘贴简历内容 |
| 简历包含敏感信息 | 提醒用户脱敏 |
| 文本过长 | 提示用户保留岗位职责、任职要求和核心经历 |

### 13.2 AI 输出异常

| 场景 | 处理方式 |
| --- | --- |
| 输出过泛 | 提供“重新生成更具体建议”入口 |
| 输出疑似编造 | 标记风险并要求用户确认 |
| 无法判断匹配度 | 提示用户补充 JD 或简历信息 |
| 改写过长 | 提供“压缩到简历 bullet”选项 |

### 13.3 产品边界

- 产品只提供简历优化建议，不保证获得面试或 offer
- 产品不能替代用户真实经历积累
- 产品不会自动投递岗位
- 产品不应保存用户敏感信息，除非用户明确授权

## 14. 后续迭代方向

### V1.1：多岗位版本对比

支持用户针对不同 JD 生成不同简历版本，并比较侧重点差异。

### V1.2：面试追问生成

根据优化后的项目经历，生成面试官可能追问的问题，帮助用户准备面试。

### V1.3：作品集优化

针对产品经理求职者的项目作品集，提供结构诊断和表达优化建议。

### V1.4：求职进度管理

支持用户记录投递岗位、简历版本、反馈状态和后续动作。

## 15. PRD 写作说明

写 PRD 时，不要只罗列功能，而要遵循以下逻辑：

1. 先写清楚为什么做：背景、用户、痛点
2. 再写清楚做什么：产品定位、目标、MVP 范围
3. 然后写清楚怎么做：流程、功能需求、页面说明
4. 最后写清楚怎么判断好不好：AI 规则、数据指标、异常状态、风险边界

对于 AI 产品，PRD 还必须额外说明：

- 输入是什么
- 输出是什么
- 输出结构是什么
- 模型不能做什么
- 如何避免幻觉、虚构和过度包装
- 用户如何判断 AI 建议是否可信
